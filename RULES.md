# RULES（当前阶段实践规则）

## 1) 资源层级规则
- 本地没有的层级，不要让 viewer 去请求。
- 默认优先保证单层可用（当前是 l3），再考虑多层优化。
- scene 的 XML 声明必须与本地库存一致。

## 2) 路径映射规则
- 以 krpano 请求 URL 为“事实标准”，后端做单向稳定映射。
- 所有重写逻辑都要可解释、可日志追踪。
- 新增映射时先在 debug 场景验证，再推广到实景。

## 3) 性能规则
- Tile/preview 图片必须允许缓存（至少短期 max-age）。
- JSON 配置接口保持 no-store，避免调试时读到旧状态。
- 先解决“重复请求同图”问题，再调并发与预取参数。

## 4) 调试规则
- 先看 Network：有没有 404、取消请求、重试风暴。
- 再看场景声明：level、tilesize、cube url 模板是否匹配本地文件。
- 最后再怀疑“资源太大”或“机器性能不够”。

## 5) 变更与回归规则
- 每次改 viewer/路由后，至少回归三类场景：
  - debug 合成场景（网格正确性）
  - 一个常用实景（稳定性）
  - 随机场景（泛化）
- 保留阶段文档（phase0_x）记录“问题-根因-修复-结论”，避免重复踩坑。
